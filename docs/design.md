# 設計方針

このドキュメントでは、プロジェクトの設計方針を記録します。

仕様の詳細は `spec.md`、作業履歴は `progress.md` に記録します。

---

## プロジェクト概要

### プロジェクト名

`Unity Support Kit`

### 目的

このプロジェクトは、以下を目的とします。

* Unityプロジェクト内の画像・BGM・SEなどのアセットを探しやすくする
* アセットの配置場所や命名規則のミスを確認しやすくする
* チーム制作における素材管理の負担を減らす
* 将来的に、Prefab・Scene・ScriptableObjectなどの管理にも拡張できる土台を作る

### 想定利用者

* Unityでゲーム制作を行う開発者
* チーム制作で素材を管理するメンバー
* 画像・音楽・効果音などのアセットを頻繁に扱う制作者

---

## 基本方針

このプロジェクトでは、以下の方針で実装します。

* 小さく作り、動作確認しながら拡張する
* 仕様・設計・進捗をドキュメントに残す
* 中核処理とUIをできるだけ分離する
* データ構造を先に安定させる
* 機能追加よりも、検索・表示・検証の安定を優先する
* 将来的な拡張を考慮するが、初期段階で過剰設計しない
* 迷った判断はこのドキュメントに記録する

---

## アーキテクチャ方針

### 中核処理

中核処理では、以下を扱います。

* アセット情報のデータ構造
* アセット検索処理
* アセット種別の判定
* パス情報の管理
* 命名規則の検証処理
* 配置フォルダの検証処理
* 使用中・未使用の簡易チェック
* 共通ロジック

中核処理では、以下に依存しすぎないようにします。

* 特定のEditorWindow
* 特定の表示形式
* 一時的なUI実装
* 特定のプロジェクト構成

理由は以下です。

* 後からUIを変更しやすくするため
* アセット情報の扱いを長く使える形にするため
* 検索・検証処理をテストしやすくするため
* 他のUnityプロジェクトでも再利用しやすくするため

## データ設計方針

データは、できるだけ明確な構造で扱います。

基本方針:

* 識別子は一意にする
* 表示名と内部IDを分ける
* 状態と設定値を混ぜない
* 実行時に変わる値と保存する値を分ける
* 接続・参照・依存関係は一か所に集約する
* 後から検証しやすい形にする

このツールでは、アセットごとに以下の情報を扱う予定です。

* アセット名
* アセット種別
* アセットパス
* GUID
* メモ
* 検証結果
* 使用中・未使用の状態

初期段階では、保存するデータを増やしすぎず、まずはUnityの `AssetDatabase` から取得できる情報を中心に扱います。

## 責務分離

各機能の責務は、以下のように分けます。

| 領域      | 責務                                  |
| ------- | ----------------------------------- |
| Core    | アセット情報のデータ構造、検索処理、検証処理、共通ロジック       |
| Runtime | 初期段階では基本的に扱わない                      |
| Editor  | EditorWindow、一覧表示、検索UI、フィルターUI、選択操作 |
| Assets  | アセット種別判定、パス管理、参照解決、プレビュー取得          |
| Docs    | 仕様、設計、進捗の記録                         |

初期段階では、すべてを作り込まず、必要な順に実装します。

---

## 対象範囲

現在のバージョンでは、以下を扱います。

* 画像アセット
* BGMアセット
* SEアセット
* アセット一覧表示
* アセット名検索
* アセット種別フィルター
* アセットパスの表示
* 選択したアセットをUnity上で確認する機能
* 画像アセットの簡易プレビュー
* 配置フォルダの簡易チェック
* 命名規則の簡易チェック

現在のバージョンでは、以下は扱いません。

* Prefab管理
* Scene管理
* ScriptableObject管理
* Addressablesを含む高度な参照解析
* 完全な未使用アセット検出
* ノベルゲーム制作専用機能
* 実行時のゲームシステム

対象外の機能は、必要になった段階で再検討します。

---

## 拡張方針

将来的には、以下の拡張を想定します。

* Prefabの一覧表示と命名規則チェック
* Sceneの一覧表示とビルド設定確認
* ScriptableObjectの一覧表示
* アセットごとのメモ保存機能
* 使用中・未使用の簡易チェック精度の向上
* 画像・音声アセットの詳細情報表示
* カテゴリ分け機能
* ノベルゲーム制作向けの素材管理機能
* 背景・立ち絵・BGM・SEをまとめて確認できる機能

ただし、初期段階では以下を優先します。

1. アセット情報のデータ構造を決める
2. Unity Editor上で専用ウィンドウを表示する
3. 画像・BGM・SEを一覧表示する
4. 検索・フィルターを実装する
5. 選択したアセットをUnity上で確認できるようにする
6. 配置フォルダや命名規則の簡易チェックを追加する
7. 必要に応じてメモ機能や未使用チェックを追加する

## 判断ログ

設計上の判断は、ここに記録します。

### 2026-06-21 初期対象アセットの決定

#### 決定

* 最初の対象アセットは、画像・BGM・SEに絞る
* Prefab・Scene・ScriptableObjectは初期段階では対象外にする
* 最初のPoCでは、一覧表示・検索・フィルター・選択を優先する

#### 理由

* 最初から対象範囲を広げすぎると実装量が増えるため
* 画像・BGM・SEは素材管理ツールとして目的が明確なため
* EditorWindowとAssetDatabaseの基本機能を学びながら実装しやすいため
* チーム制作でも素材を探しやすくする効果が分かりやすいため

#### 採用しなかった案

* 最初からPrefab・Scene・ScriptableObjectまで対応する案
* ノベルゲーム制作向けに特化した支援ツールとして作り始める案
* 未使用アセットの完全検出を初期機能に含める案

#### メモ

* 未使用チェックは便利だが、完全な検出は難しいため、初期段階では簡易チェックとして扱う
* 詳細情報の内容は、実装しながら必要なものを追加する
* メモ機能は便利だが、保存形式を決める必要があるため、初期PoC後に検討する

## メモ

設計や仕様で迷ったことは、ここに記録します。

* 初期段階では、完璧な管理ツールを目指さず、まずは「探す」「絞る」「選択する」を安定させる
* 未使用チェックは、Scene・Prefab・ScriptableObject・Addressables・コード上の文字列参照などを考慮すると難しくなる
* まずはUnityの `AssetDatabase` を利用して、Editor拡張として実装する
* アセットの詳細情報は、画像ならサイズ、音声なら長さなど、種別ごとに必要な情報が異なる
* 将来的にノベルゲーム制作向けへ拡張する場合は、背景・立ち絵・BGM・SEをまとめて扱えるカテゴリ設計が必要になる
