## 基本方針

このプロジェクトでは、以下の方針で実装します。

* 小さく作り、動作確認しながら拡張する
* 仕様・設計・進捗をドキュメントに残す
* 中核処理とUIをできるだけ分離する
* データ構造を先に安定させる
* 機能追加よりも、保存・読み込み・検証の安定を優先する
* 将来的な拡張を考慮するが、初期段階で過剰設計しない
* 迷った判断はこのドキュメントに記録する

## アーキテクチャ方針

### 中核処理

中核処理では、以下を扱います。

* `データ構造`
* `保存・読み込み`
* `検証処理`
* `共通ロジック`

中核処理では、以下に依存しすぎないようにします。

* 特定のUI
* 特定のゲームエンジン
* 特定の外部サービス
* 一時的な実装都合

理由は以下です。

* 後からUIや実行環境を変更しやすくするため
* データ構造を長く使える形にするため
* テストしやすくするため
* 複数プロジェクトで再利用しやすくするため

---

## データ設計方針

データは、できるだけ明確な構造で扱います。

基本方針:

* 識別子は一意にする
* 表示名と内部IDを分ける
* 状態と設定値を混ぜない
* 実行時に変わる値と保存する値を分ける
* 接続・参照・依存関係は一か所に集約する
* 後から検証しやすい形にする

---

## 責務分離

各機能の責務は、以下のように分けます。

| 領域      | 責務               |
| ------- | ---------------- |
| Core    | データ構造、保存・読み込み、検証 |
| Runtime | 実行、再生、反映処理       |
| Editor  | 編集UI、操作、表示       |
| Assets  | 素材管理、参照解決        |
| Docs    | 仕様、設計、進捗の記録      |

初期段階では、すべてを作り込まず、必要な順に実装します。
